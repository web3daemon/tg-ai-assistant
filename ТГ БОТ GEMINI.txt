Техническое задание: персональный Telegram-бот с AI через OpenRouter

1. Краткое описание проекта
Разработать персонального Telegram-бота для личного использования в качестве AI-ассистента для консультаций, анализа текстов, документов и изображений.

Бот работает как интерфейс между Telegram и AI-моделью, вызываемой через OpenRouter. Модель выбирается пользователем самостоятельно и задается через переменные окружения.

Первый этап разработки ориентирован на локальный запуск на одной машине без VPS, VDS, proxy и webhook. Режим работы бота: polling.

Проект разрабатывается так, чтобы в дальнейшем можно было без серьезной переработки:
- сменить модель в `.env`
- добавить голосовые сообщения
- при необходимости перенести запуск на сервер

2. Целевая платформа и стек технологий
Платформа:
Telegram Bot API.

Язык разработки:
Python 3.10+.

Рекомендуемый стек:
- `aiogram 3.x` для асинхронного Telegram-бота
- `SQLAlchemy 2.x` для работы с SQLite
- `httpx` для запросов к OpenRouter API
- `pydantic-settings` или аналог для конфигурации
- `Pillow` для базовой обработки изображений
- `PyMuPDF` и/или `pypdf` для извлечения текста из PDF
- `python-docx` для чтения `.docx`
- `openpyxl` для чтения `.xlsx`

База данных:
SQLite для хранения истории диалогов, метаданных сообщений и служебного состояния.

Размещение на первом этапе:
локальный запуск на Windows-машине пользователя.

3. Основная архитектура
Бот должен иметь модульную структуру:
- модуль Telegram-обработчиков
- модуль авторизации по whitelist
- модуль работы с БД
- модуль работы с OpenRouter
- модуль подготовки контента для модели
- модуль конфигурации

Вызов AI должен быть абстрагирован через отдельный клиент, чтобы в будущем можно было заменить провайдера или модель без переписывания обработчиков Telegram.

4. Функциональные требования
4.1. Интеграция с AI через OpenRouter
Бот должен подключаться к OpenRouter API и отправлять запросы в модель, указанную в `.env`.

Необходимо поддержать:
- имя модели через переменную окружения
- temperature через переменную окружения
- max_tokens через переменную окружения
- system prompt через переменную окружения или отдельный конфигурационный блок

Модель не должна быть зашита в коде.

На первом этапе выбирается одна модель, заданная в `.env`. Переключение модели через Telegram-команды не требуется.

4.2. Управление доступом
Бот предназначен только для одного пользователя.

Доступ должен быть ограничен по Telegram `user_id`.

Если сообщение пришло не от разрешенного пользователя, бот должен полностью его игнорировать и ничего не отвечать.

Дополнительно рекомендуется проверять:
- `chat_id`
- тип чата

Лучший вариант для текущей задачи:
бот работает только в личном чате с владельцем, а история хранится отдельно для конкретного `chat_id`. Это проще, безопаснее и лучше соответствует персональному сценарию.

4.3. Память диалога
История переписки должна храниться в SQLite бессрочно до явной очистки.

При каждом новом запросе в модель бот должен формировать контекст из:
- system prompt
- ограниченного количества последних сообщений из истории
- текущего пользовательского сообщения

Для практичности и контроля стоимости в модель следует передавать не всю историю, а последние N сообщений. При этом сама история в БД хранится полностью.

Рекомендуемая логика:
- хранить всю историю в БД бессрочно
- в запрос к модели включать только последние N сообщений
- значение N настраивать через `.env`

Нужна команда `/clear`, которая очищает историю текущего чата.

4.4. Поддержка текстовых сообщений
Бот должен принимать обычные текстовые сообщения и отправлять их в модель с учетом контекста.

Ответ модели должен сохраняться в БД и отправляться пользователю в Telegram.

Если ответ слишком длинный для одного сообщения Telegram, бот должен корректно разбивать его на несколько сообщений.

4.5. Поддержка изображений
Бот должен уметь принимать:
- обычные фото из Telegram
- изображения, отправленные как файл

Изображение должно подготавливаться для отправки в модель через OpenRouter в совместимом формате.

Изображение анализируется как часть текущего запроса. Хранить полноценную “визуальную память” в истории между сообщениями не требуется. Достаточно сохранить запись о факте сообщения и текстовый результат ответа.

4.6. Поддержка документов
Бот должен принимать и обрабатывать документы форматов:
- `.txt`
- `.pdf`
- `.docx`
- `.xlsx`

Правила обработки:
- `.txt`: читать как текст
- `.pdf`: извлекать текст из документа
- `.docx`: извлекать текст из документа
- `.xlsx`: извлекать данные по листам в текстовом виде, пригодном для передачи в модель

Документ используется как часть текущего запроса. Полное содержимое файла не обязано храниться в БД, если это избыточно. Достаточно сохранить:
- тип файла
- имя файла
- краткие метаданные
- текст пользовательского запроса
- текст ответа модели

Если документ слишком большой, должна применяться защитная логика:
- ограничение на размер файла
- ограничение на объем извлеченного текста
- понятное сообщение пользователю при превышении лимитов

4.7. Поддержка голосовых сообщений
Голосовые сообщения выносятся во второй этап разработки.

В первый релиз поддержка голосовых не входит.

Архитектура должна позволять добавить это позже без переделки ядра.

Предпочтительный вариант на второй этап:
локальное распознавание речи, чтобы не зависеть от отдельного внешнего API без необходимости.

5. Системный промпт
У бота должен быть фиксируемый system prompt.

Он задает роль AI-ассистента и стиль ответов. Значение должно настраиваться через `.env` или отдельный конфигурационный параметр.

Это позволит использовать бота как персонального помощника для:
- консультаций
- анализа документов
- разбора таблиц
- вопросов по изображениям

6. Команды бота
Минимальный набор команд первого этапа:
- `/start` — краткая информация о боте
- `/help` — список возможностей и ограничений
- `/clear` — очистка истории текущего чата
- `/status` — показать текущую модель, статус доступа и основные лимиты

Команды переключения модели на первом этапе не требуются.

7. Процесс обработки сообщений
7.1. Текст
Пользователь отправляет текст.

Бот:
- проверяет `user_id` и допустимость чата
- загружает последние N сообщений из БД
- формирует запрос к OpenRouter
- получает ответ модели
- сохраняет запрос и ответ в БД
- отправляет ответ в Telegram

7.2. Фото
Пользователь отправляет фото с подписью или без подписи.

Бот:
- проверяет доступ
- скачивает файл
- подготавливает изображение
- формирует мультимодальный запрос
- получает ответ
- сохраняет факт запроса и ответ
- отправляет результат пользователю

7.3. Документ
Пользователь отправляет документ и при необходимости сопровождающий текст.

Бот:
- проверяет доступ
- определяет тип файла
- извлекает текст или табличные данные
- при необходимости сокращает объем контента до допустимого лимита
- формирует запрос к модели
- получает ответ
- сохраняет запрос и ответ
- отправляет результат пользователю

8. Работа с ответами
Бот должен уметь:
- отправлять обычные текстовые ответы
- разбивать длинные ответы на несколько сообщений
- при необходимости формировать текстовый файл с ответом, если объем слишком большой для удобной отправки в чат

Решение о том, когда отправлять ответ сообщением, а когда файлом, задается через простой порог по длине текста.

9. Требования к конфигурации
В репозитории обязательно должен быть файл `.env.example`.

Минимально ожидаемые переменные окружения:
- `TELEGRAM_BOT_TOKEN`
- `ALLOWED_TELEGRAM_USER_ID`
- `ALLOWED_CHAT_ID`
- `OPENROUTER_API_KEY`
- `OPENROUTER_MODEL`
- `OPENROUTER_BASE_URL`
- `MODEL_TEMPERATURE`
- `MODEL_MAX_TOKENS`
- `CONTEXT_MESSAGES_LIMIT`
- `SYSTEM_PROMPT`
- `SQLITE_PATH`
- `MAX_FILE_SIZE_MB`
- `MAX_EXTRACTED_TEXT_CHARS`
- `LONG_RESPONSE_AS_FILE_THRESHOLD`

10. Требования к надежности и удобству
Бот должен:
- корректно логировать ошибки
- не падать от неподдерживаемого файла
- давать понятное сообщение при ошибке обработки
- корректно переживать перезапуск без потери истории

Временные файлы должны удаляться после обработки.

11. Требования к коду
Код должен быть:
- структурированным
- читаемым
- с понятными именами модулей
- с минимально необходимыми комментариями

Желательно предусмотреть:
- `README.md` с инструкцией по локальному запуску
- `.env.example`
- `requirements.txt` или `pyproject.toml`
- базовую структуру каталогов проекта

12. Этапы разработки
Этап 1. Базовый рабочий релиз
- локальный запуск
- polling
- whitelist для одного пользователя
- текстовые сообщения
- system prompt
- OpenRouter
- SQLite
- память диалога
- команда `/clear`
- команда `/status`
- разбиение длинных ответов

Этап 2. Расширение контента
- обработка изображений
- обработка `.pdf`
- обработка `.txt`
- обработка `.docx`
- обработка `.xlsx`
- отправка очень длинных ответов файлом

Этап 3. Голосовые сообщения
- прием голосовых
- локальное распознавание речи
- отправка расшифровки в модель

13. Критерии приемки
Бот отвечает только владельцу.

Бот работает локально без VPS, proxy и webhook.

Бот использует модель, заданную через `.env`, и ходит в OpenRouter.

Бот помнит историю разговора между перезапусками.

Бот корректно очищает историю по команде `/clear`.

Бот корректно обрабатывает текстовые сообщения.

На втором этапе бот корректно анализирует:
- PDF
- TXT
- DOCX
- XLSX
- изображения

На третьем этапе бот корректно обрабатывает голосовые сообщения.

В проекте есть чистая структура кода и инструкция по запуску.

14. Примечание по выбору модели
Выбор конкретной модели не фиксируется в ТЗ и определяется пользователем самостоятельно.

Проект должен быть совместим с заменой модели через `.env` без изменения кода, при условии что выбранная модель OpenRouter поддерживает нужный тип входа:
- текст
- изображения, если включен этап мультимодальности

При старте проекта допускается использовать одну модель для текста и другую модель позже, если это окажется выгоднее по цене или качеству.
